# AGENTS.md

## Project Goal

Build a production-ready Python service that runs in a single container and translates webhook payloads generated by **SUSE Observability** (`suse obs`) into **Mattermost** messages.

The reason this service exists is that the out-of-the-box integration cannot successfully send usable Mattermost messages because the payload fields are incompatible. This project must act as a translation layer between the incoming SUSE Observability webhook and the outgoing Mattermost message.

The final result must be suitable for deployment on Kubernetes.

---

## Primary Objective

Create a Python application that:

1. Exposes an HTTP endpoint to receive webhooks from SUSE Observability.
2. Validates and parses the incoming payload.
3. Transforms the payload into a Mattermost-compatible message.
4. Sends the translated message to Mattermost.
5. Supports health checks for Kubernetes.
6. Supports configurable message formatting.
7. Includes tests, debug logging, containerization, and CI/CD for GitHub and Docker image publishing.

## Architecture Requirements

### High-Level Design

Implement a small HTTP service with the following flow:

1. Receive webhook from SUSE Observability
2. Parse and validate the payload
3. Map payload data into an internal normalized alert model
4. Render the Mattermost message from configuration/template
5. Send the message to Mattermost
6. Return a clear HTTP response to the caller
7. Expose health/readiness endpoints for Kubernetes

### Recommended Internal Components

Use a modular design with clear separation of concerns. Suggested modules:

- `app.py` or `main.py` — application bootstrap
- `api/` — HTTP routes / request handlers
- `models/` — request and internal domain models
- `services/parser.py` — parse SUSE Observability webhook into normalized model
- `services/formatter.py` — render Mattermost message from template/config
- `services/mattermost.py` — send message to Mattermost endpoint
- `services/health.py` — health/readiness support
- `config.py` — central configuration handling
- `logging_config.py` — logging setup
- `tests/` — unit tests

You may adjust the layout, but keep the same separation of responsibilities.

## Functional Requirements

### 1. Webhook Receiver

Implement an HTTP POST endpoint to receive SUSE Observability webhooks.

Requirements:

- Accept JSON payloads
- Validate content type
- Validate required fields using request models
- Return meaningful status codes for invalid input
- Log request handling at debug level without leaking secrets

### 2. Payload Parsing

The service must translate the SUSE Observability webhook into a normalized internal representation.

Requirements:

- Use the `spec` directory to determine actual payload shape
- Extract alert/error details
- Extract affected resource details
- Extract SUSE Observability server details, including URL if available
- If the source URL is not provided directly in the payload, support building it from configuration if the spec allows a predictable base URL

### 3. Mattermost Message Rendering

Transform the normalized model into Mattermost-compatible rich text.

Requirements:

- Rich text formatting must be used
- Message must be easy to change through configuration
- Avoid hardcoding the final message body in business logic
- Prefer a configurable template-based approach

Recommended approach:

- Store the message template in configuration or a separate template file
- Support placeholders such as:
  - `{{ error_details }}`
  - `{{ resource_name }}`
  - `{{ resource_type }}`
  - `{{ resource_identifier }}`
  - `{{ suse_obs_name }}`
  - `{{ suse_obs_url }}`
  - `{{ severity }}`
  - `{{ status }}`
  - `{{ summary }}`

Use a safe templating approach. Keep it simple and deterministic.

### 4. Mattermost Delivery

Send the rendered message to Mattermost.

Requirements:

- Mattermost URL must be configurable
- Mattermost channel must be configurable
- Handle delivery failures gracefully
- Log request/response metadata at debug level where safe
- Do not log secrets or webhook tokens

### 5. Health Checks

Expose Kubernetes-compatible endpoints.

Minimum endpoints:

- `/healthz` for liveness
- `/readyz` for readiness

Behavior:

- `healthz` should confirm the application process is healthy
- `readyz` should confirm the app is ready to serve traffic
- Keep readiness lightweight unless a stronger requirement exists in the spec
- Do not make readiness depend on unstable external systems unless clearly justified

### 6. Logging and Troubleshooting

Provide structured logging with configurable verbosity.

Requirements:

- Support normal and debug logging modes
- Log incoming request correlation data where possible
- Log parsing failures
- Log outbound Mattermost delivery attempts and failures
- Mask secrets in logs
- Make logs useful for Kubernetes/container troubleshooting

Suggested config:

- `LOG_LEVEL=INFO|DEBUG|WARNING|ERROR`

### 7. Configuration

Use environment variables as the default configuration mechanism for container/Kubernetes compatibility.

Required configuration support:

- Mattermost URL
- Mattermost channel
- Log level
- Message template or template path
- Optional SUSE Observability base URL if needed to construct links
- Optional timeout settings
- Optional secret/token configuration if required by the specs

Also provide:

- Example `.env`
- Example Kubernetes manifest or Helm values snippet in the docs

## Security Requirements

Security is mandatory.

### Dependency and Image Security

- Choose dependencies with low security risk
- Do not include unnecessary packages
- Pin dependencies appropriately
- Use a slim base image
- Run as a non-root user in the container
- Avoid shell-heavy container logic
- Minimize final image size and attack surface

### CVE Requirement

- The solution should not contain **high CVEs**
- The GitHub Actions workflow should include at least one container/dependency security scan step
- Fail the pipeline on high severity findings if practical

Preferred scan examples:

- `pip-audit` for Python dependencies

### Secret Handling

- Do not hardcode secrets
- Read secrets from environment variables or Kubernetes secrets
- Do not print secrets in logs or exceptions

### Input Handling

- Validate incoming payloads
- Reject malformed requests cleanly
- Avoid arbitrary template execution or unsafe evaluation

## Message Formatting Expectations

The Mattermost message should be human-readable and structured.

At minimum, it must clearly present:

- What failed
- Which resource is affected
- Where the alert came from
- A direct URL to the SUSE Observability server or relevant alert/source

### Example Message Structure

This is an example structure only. Adapt field names based on the actual specs.

```text
**SUSE Observability Alert**

**Error**
- Summary: <summary>
- Severity: <severity>
- Status: <status>
- Details: <error_details>

**Affected Resource**
- Name: <resource_name>
- Type: <resource_type>
- Identifier: <resource_identifier>

**Source**
- Server: <suse_obs_name>
- URL: <suse_obs_url>
```

The actual message must be configuration-driven rather than hardcoded.

## Implementation Plan

Follow this order:

1. Inspect the spec directory
2. Identify actual SUSE Observability webhook schema
3. Identify Mattermost message/API requirements
4. Record assumptions only where the specs are silent
5. Create the internal normalized alert model
6. Define all fields needed for message rendering
7. Build the webhook receiver (input validation, routing, error responses)
8. Implement translation/parsing logic (convert source payload into normalized model)
9. Implement configurable message rendering (template-based output, rich text support)
10. Implement Mattermost client (configurable URL and channel, timeout handling, safe logging)
11. Add health endpoints (liveness, readiness)
12. Add logging and configuration (startup validation, debug mode)
13. Write unit tests (core business logic first, endpoint tests second)
14. Add containerization (Dockerfile, non-root user, minimal runtime image)
15. Add GitHub Actions (test, scan, build, publish)
16. Final documentation pass; make local run and deployment straightforward
